# Vapi assistants config-as-code

## Goal

Keep Vapi Assistant configuration in **version control** by exporting the canonical Assistant JSON from Vapi into `vapi/assistants/`, and syncing edited JSON back to Vapi via API.

## Repository conventions

- Source of truth in repo: `vapi/assistants/*.json`
- Index file: `vapi/assistants/index.json` (generated by export)
- Scripts:
  - `npm run vapi:export` → exports all assistants into `vapi/assistants/`
  - `npm run vapi:sync` → PATCH-updates assistants in Vapi from local JSON files

## Prerequisites

- `VAPI_API_KEY` must be set (server-side private key; never expose to the browser).
- Load envs from `.env.local` for local runs.

## Export workflow (pull from Vapi → write JSON files)

1. Ensure `VAPI_API_KEY` is present in `.env.local`.
2. Run:

```bash
set -a && source .env.local && set +a
npm run -s vapi:export
```

3. Verify outputs exist:
   - `vapi/assistants/index.json`
   - `vapi/assistants/<assistant-name>__<assistant-id>.json`

## Sync workflow (push local JSON → update Vapi)

### Safe first pass (dry run)

```bash
set -a && source .env.local && set +a
npm run -s vapi:sync -- --dry-run
```

### Apply updates

```bash
set -a && source .env.local && set +a
npm run -s vapi:sync
```

## Notes / gotchas

- Sync requires `assistant.id` to exist in each JSON file. Exported files include it.
- Sync uses `PATCH /assistant/{id}` and strips common server-managed fields (`id`, `orgId`, `createdAt`, `updatedAt`) from the request body defensively.
- Client-side keys like `NEXT_PUBLIC_VAPI_PUBLIC_KEY` are unrelated to the server API key used by these scripts.
